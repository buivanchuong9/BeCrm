# DermaHealth Backend ‚Äî Frontend Migration Map

Maps every frontend mock/localStorage source to its real backend replacement, updated as each backend module ships. Per spec section 38/"FE-BE contract governance": a frontend screen must use either mock data or API data during migration, never merge both as authoritative in the same screen. **No frontend code has been touched yet** ‚Äî T19‚ÄìT21 (frontend integration) have not started; this document records the mapping in advance so the eventual FE migration has a ready reference, and confirms which pieces are now safe to start planning against.

## Ready for FE migration (T03 delivered)

| Frontend source | Current behavior | Replacement | Required FE change |
|---|---|---|---|
| `src/state/AppStateContext.tsx` ‚Äî `SESSION_KEY = 'dermahealth:v1:session:currentUserId'`, `DEFAULT_USER_ID = 'U-0001'` | Reads/writes a raw user ID string to `localStorage`; no credential check anywhere. | `POST /api/v1/auth/sessions` (real Argon2id verification) + `POST /api/v1/auth/session-refreshes` (silent session restore) + `DELETE /api/v1/auth/sessions/current` (logout) / `DELETE /api/v1/auth/sessions` (logout all devices). *(Renamed from the initial `/auth/login`/`/auth/refresh`/`/auth/logout` during the docs/api.md contract realignment ‚Äî see API_CHANGELOG.md's "T03 contract realignment" entry.)* | Replace the `localStorage` read in `AppStateProvider` with an in-memory access-token store (never localStorage ‚Äî spec section 28) plus a refresh call on app boot. The `refresh_token` cookie is `HttpOnly`, so FE cannot read it directly; it is sent automatically by the browser to `/api/v1/auth/*` when `credentials: 'include'` is set on fetch/XHR. |
| `src/pages/Login.tsx` ‚Äî `onFinish={() => nav('/app/dashboard')}`, form values never read | Zero credential validation; navigates unconditionally. | `POST /api/v1/auth/sessions` with real `{email, password, rememberMe}` body. | Wire `Form.onFinish` to call the login endpoint; handle `401 AUTH_INVALID_CREDENTIALS` / `AUTH_ACCOUNT_LOCKED` with a generic user-facing message (do not distinguish "wrong password" from "unknown email" in the UI either, matching backend behavior). Remove the hardcoded `initialValues={{email:'nguyenvana@gmail.com', password:'password123'}}` ‚Äî seeded dev accounts now use `@example.test` emails (see `prisma/seed/index.ts`) and a value from `SEED_DEMO_PASSWORD`, never a value baked into frontend source. |
| `src/layouts/TopHeader.tsx` ‚Äî account-switcher `<Select>` calling `setCurrentUserId(userId)` directly | Lets any user impersonate any seeded identity with one click, bypassing login entirely. | No backend equivalent ‚Äî intentionally not implemented. | This must become a development-only feature gated behind an explicit build flag (spec section 38: "dev role switch only behind explicit development endpoint/build flag"), or be removed. Do not port this to a flag that could ship to production. |
| `src/domain/guards.ts` ‚Äî `assertRole(actorId, allowed: UserRole[])`, single flat role check, no relationship/ownership logic | Client-side, trivially bypassable (`localStorage` write). | `RolesGuard` (`src/common/authorization/roles.guard.ts`) server-side, applied via `@Roles(...)`; resource-relationship policies (`src/modules/patients/policies/patient-policies.ts`) for anything ownership-scoped. | FE role gates (`RoleProtectedRoute`, `hasRoleAccess`) may stay as UX-only hiding (spec: "Route hiding is never authority") but must stop being treated as security ‚Äî every request now needs a valid Bearer token regardless of what the FE UI shows/hides. |
| `src/domain/seed.ts` ‚Äî `users` array, `U-0014`/`U-0015` duplicated | 19 array entries, 2 ID collisions, no org/clinic FK, `department` is a free string. | `prisma/seed/index.ts` ‚Äî 19 unique users, 1 organization (`dermahealth`), 1 clinic (`CS-HCM-01`), 12 departments, `UserMembership` rows with real FKs. | When FE fetches `/api/v1/me`, `memberships[].departmentId` is a UUID, not a display string ‚Äî FE resolves department *names* via `GET /api/v1/departments?clinicLocationId=...` rather than assuming `department` is human-readable text. |
| `src/domain/services/patientService.ts` ‚Äî `getCurrentPatient()` (`patientRepository.getAll()[0]`, explicitly documented as a single-tenant hack), `getPatient(id)`, `getPrimaryDoctor(patient)` | Client-side in-memory lookups against the seeded `Patient[]` array; "current patient" is just array index 0, not derived from any session. | `GET /api/v1/patients` (patient role ‚Üí forced to own single record, replacing the `[0]` hack ‚Äî see decision D-016) for "current patient"; `GET /api/v1/patients/{patientId}` for detail/primary-doctor projection (`primaryDoctor: {id,name}` already resolved server-side, no separate lookup call needed). | Remove the `[0]`-index assumption entirely; call the list endpoint once on patient-role bootstrap and cache the single returned record. `dob` arrives as canonical `YYYY-MM-DD`, not `DD/MM/YYYY` ‚Äî update any direct string rendering in `Profile.tsx`. |
| `src/domain/services/patientService.ts` ‚Äî `setConsent(patientId, type, granted)`; `src/pages/SettingsPage.tsx` ‚Äî `CONSENT_LABEL` map, immediate-write `onChange` toggle | Single upsert-style function flips `granted` and stamps `grantedAt`/`withdrawnAt` on the same repository row; no history, no `policyVersion`. | `POST /api/v1/patients/{patientId}/consent-grants` (grant) and `POST /api/v1/patients/{patientId}/consent-withdrawals` (withdraw) ‚Äî separate append-only-backed endpoints, both requiring `Idempotency-Key` and (withdrawal) the current `version`. | Split the single `setConsent` call into two endpoint calls by direction; add an `Idempotency-Key` header (e.g. a UUID per toggle interaction) and pass the last-known `version` on withdrawal ‚Äî fetch `GET .../consents` first if the FE doesn't already hold it. The FE's separate, unrelated local-state "privacy" toggle block in `SettingsPage.tsx` (2FA, "allow doctor to view history", etc.) has no backend model at all and must not be wired to these endpoints ‚Äî only the three `Consent`-entity-backed toggles (`data_processing`, `research_data_sharing`, `telemedicine`) map here. |
