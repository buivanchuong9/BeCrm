# DermaHealth Backend — Frontend Migration Map

Maps every frontend mock/localStorage source to its real backend replacement, updated as each backend module ships. Per spec section 38/"FE-BE contract governance": a frontend screen must use either mock data or API data during migration, never merge both as authoritative in the same screen. **No frontend code has been touched yet** — T19–T21 (frontend integration) have not started; this document records the mapping in advance so the eventual FE migration has a ready reference, and confirms which pieces are now safe to start planning against.

## Ready for FE migration (T03 delivered)

| Frontend source | Current behavior | Replacement | Required FE change |
|---|---|---|---|
| `src/state/AppStateContext.tsx` — `SESSION_KEY = 'dermahealth:v1:session:currentUserId'`, `DEFAULT_USER_ID = 'U-0001'` | Reads/writes a raw user ID string to `localStorage`; no credential check anywhere. | `POST /api/v1/auth/login` (real Argon2id verification) + `POST /api/v1/auth/refresh` (silent session restore) + `POST /api/v1/auth/logout`. | Replace the `localStorage` read in `AppStateProvider` with an in-memory access-token store (never localStorage — spec section 28) plus a refresh call on app boot. The `refresh_token` cookie is `HttpOnly`, so FE cannot read it directly; it is sent automatically by the browser to `/api/v1/auth/*` when `credentials: 'include'` is set on fetch/XHR. |
| `src/pages/Login.tsx` — `onFinish={() => nav('/app/dashboard')}`, form values never read | Zero credential validation; navigates unconditionally. | `POST /api/v1/auth/login` with real `{email, password}` body. | Wire `Form.onFinish` to call the login endpoint; handle `401 AUTH_INVALID_CREDENTIALS` / `AUTH_ACCOUNT_LOCKED` with a generic user-facing message (do not distinguish "wrong password" from "unknown email" in the UI either, matching backend behavior). Remove the hardcoded `initialValues={{email:'nguyenvana@gmail.com', password:'password123'}}` — seeded dev accounts now use `@example.test` emails (see `prisma/seed/index.ts`) and a value from `SEED_DEMO_PASSWORD`, never a value baked into frontend source. |
| `src/layouts/TopHeader.tsx` — account-switcher `<Select>` calling `setCurrentUserId(userId)` directly | Lets any user impersonate any seeded identity with one click, bypassing login entirely. | No backend equivalent — intentionally not implemented. | This must become a development-only feature gated behind an explicit build flag (spec section 38: "dev role switch only behind explicit development endpoint/build flag"), or be removed. Do not port this to a flag that could ship to production. |
| `src/domain/guards.ts` — `assertRole(actorId, allowed: UserRole[])`, single flat role check, no relationship/ownership logic | Client-side, trivially bypassable (`localStorage` write). | `RolesGuard` (`src/common/authorization/roles.guard.ts`) server-side, applied via `@Roles(...)`. | FE role gates (`RoleProtectedRoute`, `hasRoleAccess`) may stay as UX-only hiding (spec: "Route hiding is never authority") but must stop being treated as security — every request now needs a valid Bearer token regardless of what the FE UI shows/hides. |
| `src/domain/seed.ts` — `users` array, `U-0014`/`U-0015` duplicated | 19 array entries, 2 ID collisions, no org/clinic FK, `department` is a free string. | `prisma/seed/index.ts` — 19 unique users, 1 organization (`dermahealth`), 1 clinic (`CS-HCM-01`), 12 departments, `UserMembership` rows with real FKs. | When FE fetches `/api/v1/me`, `memberships[].departmentId` is a UUID, not a display string — FE must resolve department *names* via a future `/organizations/{id}/departments` endpoint (not yet built) rather than assuming `department` is human-readable text. |
